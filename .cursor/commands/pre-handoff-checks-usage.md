# Pre-Handoff Checks: How to Use

**Автоматические проверки перед передачей задачи следующему агенту**

## 🚀 Как использовать

### Способ 1: Комментарий в Issue

Просто оставь комментарий в Issue с командой валидации:

```markdown
/backend-validate-result
```

**Workflow автоматически:**
- Обнаружит команду
- Найдёт Issue number
- Определит агента (backend)
- Запустит проверки

### Способ 2: Manual trigger (workflow_dispatch)

1. Зайди в Actions → Pre-Handoff Checks
2. Нажми "Run workflow"
3. Введи:
   - Issue number: `123`
   - Agent: выбери из списка

## 🔍 Что проверяется

### Backend

1. **Определение сервиса:**
   - Из Issue labels (например, `companion-service`)
   - Из Issue body (упоминание `companion-service-go`)
   - Из связанного PR (изменённые файлы)
   - Из последних коммитов

2. **Проверки:**
   - ✅ OpenAPI spec существует и валиден
   - ✅ Go код компилируется
   - ✅ Тесты проходят
   - ✅ Docker образ собирается (если есть Dockerfile)

3. **Результат:**
   - ✅ Success → комментарий с результатами
   - ❌ Failure → комментарий с ошибками
   - ⚠️ Service not found → подсказки как исправить

### UE5

- Проверка существования UE5 проекта
- Базовая проверка C++ файлов

### API Designer

- Валидация всех OpenAPI спецификаций
- Проверка использования common.yaml
- Проверка размера файлов (<500 строк)

### Database

- Проверка существования миграций
- Валидация структуры файлов миграций

## ⚠️ Важно

**Это автоматические проверки, НЕ заменяют ручную валидацию!**

1. Workflow проверяет **технические** аспекты:
   - Компиляция
   - Тесты
   - Синтаксис

2. Агент всё равно должен проверить **функциональные** аспекты:
   - Соответствие требованиям Issue
   - Acceptance criteria
   - Business logic

3. **Если автоматические проверки прошли:**
   - ✅ Техническая часть готова
   - ⚠️ Всё равно используй `/backend-validate-result` для полной проверки

4. **Если автоматические проверки провалились:**
   - ❌ Исправь ошибки
   - 🔄 Запусти проверки снова
   - ⛔ НЕ передавай задачу пока проверки не пройдут

## 🐛 Troubleshooting

### "Cannot determine service name"

**Причина:** Workflow не может автоматически определить какой сервис проверять.

**Решение:**
1. Добавь label на Issue: `companion-service` (или другой)
2. Упомяни сервис в Issue body: `companion-service-go`
3. Свяжи PR с Issue, который меняет файлы в `services/companion-service-go/`

### "OpenAPI spec not found"

**Причина:** OpenAPI спецификация не найдена.

**Решение:**
- Проверь что файл существует: `proto/openapi/{service-name}-service.yaml`
- Или используй точное имя из Issue

### "Tests failed"

**Причина:** Тесты не проходят.

**Решение:**
- Запусти тесты локально: `cd services/{service}-go && go test ./...`
- Исправь падающие тесты
- Проверь что все зависимости установлены

## 📊 Пример использования

```markdown
# В Issue комментарий:

@agent /backend-validate-result

# Workflow автоматически:
1. Обнаружил команду
2. Определил что это backend validation
3. Нашёл Issue #123
4. Определил сервис из labels: companion-service
5. Запустил проверки:
   - ✅ OpenAPI spec found
   - ✅ Go build successful
   - ✅ Tests passed
   - ✅ Docker build successful
6. Оставил комментарий с результатами

# Агент видит:
✅ Automated checks passed. Ready for handoff to Network.
```

## ✅ Best Practices

1. **Запускай автоматические проверки ДО ручной валидации:**
   - Сначала `/backend-validate-result` (automatic)
   - Потом используй команду валидации (manual checklist)

2. **Исправляй ошибки сразу:**
   - Если проверки провалились → исправь
   - Не передавай задачу с ошибками

3. **Используй labels:**
   - Добавляй service labels на Issue
   - Это помогает workflow определить сервис автоматически

4. **Проверяй результаты:**
   - Читай комментарии от workflow
   - Убедись что все проверки прошли

## 🔧 Настройка

**Workflow настраивается автоматически, но можно:**

1. **Добавить дополнительные проверки:**
   - Отредактировать `.github/workflows/pre-handoff-checks.yml`

2. **Изменить условия запуска:**
   - Текущее: при комментарии или manual trigger
   - Можно добавить: на изменение статуса, на PR creation

3. **Расширить проверки:**
   - Добавить линтеры
   - Добавить security scans
   - Добавить coverage checks
