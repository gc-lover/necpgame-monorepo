## 1. Документы .BRAIN

- `draft` — черновой материал, доступен только команде подготовки.
- `review` — ожидает проверки менеджером или профильным агентом.
- `approved` — утвержден, доступен для создания API задач.
- `deprecated` — устаревший, оставлен для истории или архива.

**Правила:**
- Перевод в `approved` возможен только после актуализации метаданных и ссылок.
- При изменении содержания версию повышаем семантически (например, 1.0.0 → 1.1.0).

---

## 2. Готовность к API (readiness-tracker.yaml)

- `ready` — документ полностью готов, ДУАПИТАСК может создавать задания.
- `needs-work` — требуется доработка содержания/структуры.
- `blocked` — нет критичных данных, ожидаем связанные материалы.
- `in-review` — документ проверяется Brain Readiness Checker.
- `not-applicable` — документ не предназначен для API (идея, служебный файл).

**Обязательные поля записи:** путь, версия, статус, приоритет, время проверки, проверяющий, целевые микросервис/директория/фронтенд модуль.
**Очереди:** каждая запись дублируется в `06-tasks/queues/<status>.md` (лимит 500 строк, далее `_0001`, `_0002`, …). При смене статуса перемещайте запись между файлами.

---

## 3. Задачи API (brain-mapping.yaml, tasks/active/queue/)

- `queued` — задача создана и ожидает взятия АПИТАСК.
- `in_progress` — исполнитель работает над заданием.
- `review` — задание ожидает ревью от менеджера/лида.
- `ready_for_spec` — можно переходить к OpenAPI (по результатам проверки).
- `completed` — OpenAPI спецификация готова и прошла валидацию.
- `archived` — перенесено в архив (после реализации и очистки).
- `cancelled` — задание отменено, указаны причины.

**Очереди:** записи по каждому статусу ведутся в `API-SWAGGER/tasks/queues/<status>.md` с лимитом 500 строк (создавайте `_0001`, `_0002`, … при переполнении).

---

## 4. Реализация backend/frontend (implementation-tracker.yaml)

### Статус API спецификации
- `queued` — ожидает создания/обновления OpenAPI.
- `in_progress` — АПИТАСК работает над спецификацией.
- `completed` — спецификация обновлена и провалидирована.
- `failed` — работа остановлена, требуется вмешательство (описать причину).

### Статусы backend/frontend
- `not_started` — реализация не начата.
- `in_progress` — команда работает над реализацией.
- `completed` — функциональность реализована и протестирована.
- `failed` — реализация остановлена/отклонена (указать причину).

Для каждого перехода обновляйте поля `started`, `completed`, `agent`, `commit`, `notes` и приоритет.
**Очереди:** backend — `BACK-GO/docs/implementation-queues/backend/<status>.md`, frontend — `FRONT-WEB/docs/implementation-queues/frontend/<status>.md`. Лимит 500 строк с продолжениями `_0001`, `_0002`, … для каждого файла.

## 5. Планирование Manager (TODO, current-status, open-questions)

### TODO.md
- `[ ]` — задача открыта.
- `[~]` — в работе.
- `[x]` — завершена и переселена в архив.
- `[!]` — блокер, требует отдельного внимания.

### current-status.md
- `active` — активные направления (с привязкой к документам).
- `pending` — ожидают входных данных.
- `blocked` — зависят от внешних решений.

### open-questions.md
- `open` — вопрос сформулирован, ответа нет.
- `answered` — есть решение, подготовить действие.
- `archived` — перенесено в `archive/resolved-questions/`.

---

## 6. Приоритеты

- `critical` — блокирует MVP/релиз, берем в работу немедленно.
- `high` — важная функциональность ближайших спринтов.
- `medium` — значимая, но не блокирующая работа.
- `low` — улучшения, исследования, резерв.

При присвоении приоритета учитывайте зависимость от других систем (auth, progression, economy и т.д.).

---

## 7. Жизненный цикл фичи

1. **Документ** → `draft` → `review` → `approved` (фиксируем версию).
2. **Готовность** → `ready` в `readiness-tracker.yaml`.
3. **Задача API** → `queued` → `in_progress` → `ready_for_spec` → `completed`.
4. **Реализация** → backend/frontend `in_progress` → `completed`.
5. **Архивация** → Clear переносит записи в `06-tasks/archive/…`.

---

## 8. Контроль изменений

- Любое изменение статуса фиксируйте актуальным временем (см. процедуры).
- После обновления статусов фиксируйте изменения вручную через git (`git add`, `git commit`, `git push`) или MCP.
- При спорных ситуациях эскалируйте через `open-questions.md` и назначайте ответственных.
