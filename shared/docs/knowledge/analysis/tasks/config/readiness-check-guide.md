# Руководство по проверке готовности документов к созданию API задач

**Версия:** 1.0.0  
**Обновлено:** 2025-11-08 23:49  
**Ответственный:** Brain Readiness Checker

---

## 1. Подготовка
1. Получите фактическое время (PowerShell команда из процедур).
2. Ознакомьтесь с документом полностью, проверьте метаданные и структуру.
3. Сверьтесь с `STATUSES-GUIDE.md` и `ЧЕКБРЕЙН.MD`.

---

## 2. Критерии готовности

| Критерий | Требования | Статус при несоответствии |
| --- | --- | --- |
| Статус документа | `review` или `approved`, версия указана | `needs-work` |
| Полнота данных | Есть сценарии, правила, примеры, нет пробелов | `needs-work` |
| Конкретика | Конкретные механики без общих фраз | `needs-work` |
| Зависимости | Указаны связанные документы и API каталоги | `blocked` при отсутствующих документах |
| TODO/риски | Нет блокирующих TODO | `blocked` |
| Согласованность | Не противоречит другим утвержденным документам | `in-review` до синхронизации |
| Версия/дата | Указаны `Версия`, `Обновлено` | `needs-work` |

---

## 3. Обновление `readiness-tracker.yaml`

Добавьте запись вида:
```yaml
- path: ".BRAIN/05-technical/backend/authentication.md"
  version: "1.0.0"
  status: "ready"
  priority: "critical"
  checked: "2025-11-08 23:49"
  checker: "Brain Readiness Checker"
  api_target:
    microservice: "auth-service"
    directory: "api/v1/auth/authentication.yaml"
    frontend_module: "modules/auth"
  notes: "Подробные кейсы аутентификации, требований хватает для API."
```

---

## 4. Связь с другими трекерами

- Статус `ready` → уведомляем Brain Manager и ДУАПИТАСК.
- Статус `needs-work` → добавляем пункт в `TODO.md` с пометкой `[!]`.
- Статус `blocked` → формируем запись в `open-questions.md` с ответственным.
- Статус `in-review` → фиксируем в `current-status.md` (раздел `pending`).
- Статус `not-applicable` → указываем причину в `notes`.

---

## 5. Коммуникация

- Для спорных моментов создаём раздел в конце документа `## Readiness Notes`.
- Поддерживаем версию документа; после доработок повторяем проверку.
- При массовой проверке используем единое время для набора документов.

---

## 6. Завершение проверки

1. Обновите `readiness-tracker.yaml`.
2. Отразите изменения в `current-status.md`.
3. При необходимости добавьте вопрос/задачу.
4. Сообщите менеджеру о готовых документах.
5. Зафиксируйте изменения вручную через git (`git add`, `git commit`, `git push`) или MCP после серии правок.
